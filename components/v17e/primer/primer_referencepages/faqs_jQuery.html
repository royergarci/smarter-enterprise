What is v17e?
v17e is the mobile optimized version of the v17 design system for ibm.com. It's a framework engineered to help you create a cohesive and engaging digital experience on the ibm.com domain. It's a foundation that you can build on top of with flexibility and efficiency. It's a set of tools and resources that will allow you to focus on what really matters: the customer.
How does v17e differ from v17?
We’ve added adaptive grids for desktop, tablet, and mobile screens, updated jquery code, converted icons into fonts, customized IBM fonts, and created new adaptive modules.
What are the biggest changes I can expect in v17e? What are the main classes?
v17e outlines several new classes and containers that you can use to improve your page appearance across devices:
  1. The ibm-resize class can be added to an image tag to allow the size of the image to adapt to changing window dimensions.
  2. The ibm-band container and ibm-band class creates an edge to edge band around ibm-columns, which both opens up the space to the left and right of the content for use, and visually breaks up the page into horizontal bands.
  3. The ibm-sprite container allows your image to look different depending on mouseover or mousedown, while employing only a single image (sprite).
How do I get started?
If you’re interested in rolling out your site using v17e, we suggest you begin by testing it out. To get started, please provide the following information to Pattern Lab Request/Somers/IBM: your site name, primary contact name and email address, approximate number of pages, and sample URL.
How can I get involved in the development process?
Contact Jeff Florentino, who will assess your skills and your site’s compatibility with the project. If you qualify, he will grant you and your development team access to the v17_enhanced SVN repository. Your developers will need to be familiar with v17 code and elements found on the CDDS site before they get set up to start coding with v17e.
Will non-mobile users benefit?
v17e improves performance across all devices. You can expect faster loading pages, web page consistency across devices, and better performance on retina screens.
How does v17e improve performance?
  1. Splitting up the CSS file to deliver only the files that are relevant to each page’s functions.
  2. Eliminating duplicate files on different ibm.com sites and combining JS files for fewer calls and better use of caching.
  3. Standardizing file compression.
  4. Optimizing the CSS code with a CSS pre-processor.
  5. Changing graphical icons to fonts.
  6. Removing code and CSS that is no longer needed (reaching back through v16).
  7. Ensuring all centrally delivered JS and CSS files are delivered via Akamai's geographically distributed CDN network of servers.
Why weren't these improvements added to v17?
If we had decided to add on to v17, mandatory changes would have to be made to 100% of ibm.com pages. Given the fact that many things were completely rewritten, we decided it would be impractical to coordinate and test a modified version of v17 all at once. Our solution was to create an enhanced version that can be implemented on a rolling basis.
Is v17e compatible with the Lite Blue mobile proxy, or will I need to override it?
v17e is replacing the Lite Blue mobile proxy, so if your site currently uses it, you will need to remove any go-mobile type .js references. To override Lite Blue, use <link rel="IBM.Mobile.redirect" href="http://www.ibm.com/[UPDATE]/" /> which will redirect users to an alternate mobile version of the page. The "[UPDATE]" portion of that URL should have the canonical URI for the page so if the user accidentally lands at m.ibm.com, they will be redirected back to www.ibm.com. You can learn more about the Lite Blue proxy and the IBM.Mobile.redirect and/or reach out to the IBM Corporate Webmasters for help.
What if I don't adopt v17e? When will it be mandatory to be on v17e?
Migration to v17e is not currently required, so you can continue to use v17 for now. The terms and timing of mandatory adoption to v17e will communicated if v17 is discontinued or deprecated. Mandatory migration is TBD. v17e is a prototype framework. Eventually it is expected that the v17e changes will be incorporated into the follow-on to v17 (maybe named v18?) and at that time the terms and timing of adoption will be developed and communicated.
How much work is involved in changing a v17 page into a v17e page?
Depending on your pages, potentially not a lot of work. For example, these would be the minimal steps that would be needed to migrate a page to v17e. To take advantage of some advanced v17e functions will require some additional work.
  1. Add the viewport meta tag
    e.g.
    <meta name="viewport" content="width=device-width, initial-scale=1"/>
  2. Change the .css file you are calling.
  3. For non-DBCS pages, add the "ibm-type" class to the page's body tag (e.g. change the body tag from <body id="ibm-com"> to <body id="ibm-com" class="ibm-type">). The "ibm-type" class applies the custom IBM fonts. This must not be applied to DBCS pages.
  4. The default template (left nav template) now requires an "ibm-default" class
    e.g.
    <div id="ibm-top"> would become <div id="ibm-top" class="ibm-default">
  5. If the page includes any of the following: a data table, list modules, form, application, and/or social icons for the content space, you'll need to add a call to the appropriate css file.

When can I start using v17e on my site?
If you qualify for our pilot/beta test, right away. You can apply to be a part of the v17e pilot/beta test by contacting Pattern Lab Request/Somers/IBM and provide the following information: site name, primary contact name and email address, approximate number of pages, and sample URL.
When will v17e roll out?
Continuously. For those enrolled in the pilot/beta test, v17e is already live. Because v17e is an agile project, you can expect the CSS and/or JS files to be published approximately every 2 weeks.
How do I get started on SVN?
If your team has been granted approval to assist with the development of the v17e code, Jeff Florentino or another administrator of SVN will grant you the appropriate access. We suggest you use the software Versions to keep track of the different versions of v17e as they are created. v17e contributors will also need a SASS compiler; you can either use the command prompt, CodeKit, or prepros.
Where are the main JS and CSS files?
They are located at v17_enhanced/trunk/css and v17_enhanced/trunk/js. The production references are: <link href="//1.www.s81c.com/common/v17e/css/ww.css" rel="stylesheet" title="www" type="text/css" /> and <script src="//1.www.s81c.com/common/v17e/js/ww.js" type="text/javascript"></script>. jQuery and Modernizr are included as part of the common v17e JS.
What are the main classes that are available in v17e?
Some of the main classes include new classes for bands, containers, and modules. They can be referenced in greater detail once you are on-boarded.
What viewports are supported under v17e?
The iphone, desktop, and the the majority of tablets (Nexus, iPad, and Galaxy) are currently supported, and we will add more viewports on an ongoing basis. Testing is as simple as accessing the dev server on your mobile device, such as Resizr, or collapsing a desktop browser window.
How do I modify or add my own custom components to v17e?
Some sites may need to have custom components to accommodate their business needs. In v17e, custom components are added through a site specific CSS file (ww_cci.css). To define the custom styles, use the '.ibm_cci-' prefix, followed by the modified module (e.g. .ibm_cci-leadspace). All custom modules must be reviewed by the v17e team prior to start of coding, and approved after coding is complete. Once approved, you can add customizations to your site specific ww_cci.css file.

What are the conditions of adopting?
The v17e team will be the point of contact for any questions, concerns, or issues throughout the process. If you are interested in being a part of the v17e pilot/beta test, send the following information to Pattern Lab Request/Somers/IBM: Site name, Primary contact name and email address, approximate number of pages, sample URL. Web sites wishing to adopt must be pre-approved by the v17e team. Teams accepted into v17e will need to agree with our terms and conditions of adopting. You must keep in mind that this work is ongoing. The initial adoption process involves migrating some sample pages to v17e to ensure your site won’t have any problems. Once you are confident that your site works well with v17e, you can request a review of pages before publishing live.Your site may be affected by changes to the v17e common library (CS / JS). Participants in the v17e pilot/beta test must agree to follow the deploy schedule and retest prior to each release to ensure that any live pages will not be adversely affected.
