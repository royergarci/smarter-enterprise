Template updates to enable v17e

What is v17e?
17e is the mobile optimized version of the v17 design system for ibm.com. It's a framework that will enable you to create a cohesive and engaging digital experience on the ibm.com domain. It's a foundation you can build on top of with extreme flexibility and efficiency. It's a set of tools and resources to help you focus on what really matters, the customer.
What's new in v17e?
Adaptive grids for desktop, tablet, and mobile screens. Other enhancements: updated jquery code, icons have been converted to fonts, IBM custom fonts, as well as new, adaptive modules.
What's new in v17e? (details)
There are a few new classes and containers in v17e that you can use to improve your page appearance for mobile devices or just generally to tweak the page. Three biggies include 1) adding the ibm-resize class to an image tag allows the image to get smaller (or larger) as needed based on window size, 2) the ibm-band class puts an edge to edge band around ibm-columns, visually breaking the space into horizontal bands and opening up the space to the left and right of the content for use, and 3) ibm-sprite lets you have an image that looks different on mouse over or down using a single image (sprite).
How do I get started? How do I get access?
Your developement team can first get started by contacting Jeff Florentino who will then grant you and your dev team access to the v17_enhanced SVN repository. Your developers should first get familiar with v17 code and elements found on the CDDS site. Once familiar, you will be set up to start coding with v17e.
Will non-mobile users benefit?
Yes, experience on desktop for retina screens has been improved.
Does v17e improve performance?
Yes. Improved performance is being delivered through a number of techniques including 1) splitting up the CSS file and only delivering the ones that are needed for the functions that are actually used on each page, 2) eliminating duplicate files on different ibm.com sites and combining different JS files so that there are fewer calls and better use of caching, 3) ensuring every file is compressed, 4) using CSS pre-processor to optimize CSS code, 5) changing graphical icons to fonts, 6) removing code and CSS that are no longer needed (including for V16) and 7) ensuring all centrally delivered JS and CSS files are delivered via Akamai's geographically distributed CDN network of servers, to name a few.
Why weren't these improvements added to v17?
Because many things have been rewritten from the ground up. Mandatory changes would have to be made to 100% of all ibm.com pages, and with the number of pages on v17 today, it would be impractical to coordinate and test all at once. For this reason it was determined that making this one mid-course correction (and adding in a large number of additional enhancements at the same time) is a more reasonable solution.
Is v17e a replacement for the 'Lite Blue' mobile proxy or can the two be used together?
Yes, v17e is a replacement for and will not be used with the mobile proxy. If your site is currently using the mobile proxy, you will need to check and remove any go-mobile-type .js references. Use <link rel="IBM.Mobile.redirect" href="http://www.ibm.com/[UPDATE]/" /> to override 'Lite Blue', causing it to redirect users to an "alternate mobile version of a page." The "[UPDATE]" portion of that URL should have the canonical URI for the page. That way, if the user accidentally lands at m.ibm.com, they will be redirected back to www.ibm.com instead. What if I don't adopt v17e?
Migration to v17e is not required. You can continue to use v17 for now.
How much work is involved in changing a v17 page into a v17e page?
  1. Add the viewport meta tag
    e.g.
    <meta name="viewport" content="width=device-width, initial-scale=1"/>
  2. Change the .css file you are calling.
  3. For non-DBCS pages, add the "ibm-type" class to the page's body tag (e.g. change the body tag from <body id="ibm-com"> to <body id="ibm-com" class="ibm-type">). The "ibm-type" class applies the custom IBM fonts. This must not be applied to DBCS pages.
  4. The default template (left nav template) now requires an "ibm-default" class
    e.g.
    <div id="ibm-top" class="ibm-default">
  5. If the page includes any of the following: a data table, list modules, form, application, and/or social icons for the content space, you'll need to add a call to the appropriate css file.

When will v17e roll out?
For the people already in pilot, v17e is live. v17e is an agile project and as such, the CSS and/or JS files are published every 2 weeks.
When will it be mandatory to be on v17e?
v17e is a prototype framework. Mandatory migration is TBD.
How do I get started on SVN?
Have your development team contact Jeff Florentino, or other administrators of SVN, to grant you the appropriate access. Suggested software to use is 'Versions', to keep track of the different versions that are created. v17e contributors will also need a SASS compiler. You can either use the command prompt, CodeKit or prepros.
Where are the main JS and CSS files?
Once you are granted access to SVN, you will then be able to download the appropriate JS and CSS files from the SVN directory. They are located at v17_enhanced/trunk/css and v17_enhanced/trunk/js. The production references are: <link href="//1.www.s81c.com/common/v17e/css/ww.css" rel="stylesheet" title="www" type="text/css" /> and <script src="//1.www.s81c.com/common/v17e/js/ww.js" type="text/javascript"></script>. jQuery and Modernizr are included as part of the common v17e JS.
What are the main classes that are available in v17e?
Some of the main classes include new classes for bands, containers, and modules. They can be referenced in greater detail once you are on-boarded.
What viewports are supported under v17e?
Tablets (Nexus, iPad, Galaxy), phones (iPhone) and desktop are supported. Testing is as simple as accessing the dev server on your mobile device. Another option to test the viewports is to collapse the desktop browser window.
How do I modify or add my own custom components to v17e?
Custom components are added through a site specific CSS file (ww_cci.css). Inside of this file you can define the custom styles by using a '.ibm_cci-' prefix followed by the modified module (e.g. .ibm_cci-leadspace). All custom modules must be reviewed and approved by the v17e team. Customizations that are approved by the v17e team can be added to your site specific ww_cci.css file.
